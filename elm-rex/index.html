Elm Compiles to JS

Elm is a language to write front-end applications

Compiles to ES3 => enjoy IE6!

Comprehensive

As a complete language, it reduces choice fatigue

A functional language

  • Functions are pure
  • Data is immutable
  • No runtime errors
  • Semver is imposed
  • A helpful compiler
  • Functions are curry-able

The 1st class citizen is the function, the aim is to reason with them easily

  • Pure = idem-potent & no side-effect
  • Immutable => a class of bugs that disappears
  • If it compiles, it runs
  • Strong typing brings two benefits: reliable semver + helpful compiler
  • The compiler knows lots about your program, it GUIDES you
  • Currying => composition

Helpful compiler

You follow the compiler, it finds almost all non-functional errors

The messages are WAY BETTER than stack traces

Currying

Currying is the process of translating the evaluation of a function that takes multiple arguments into evaluating a sequence of functions, each with a single argument. Source Wikipedia

Composition, the definition is a nightmare so let's give examples

Currying

          
            add : Int -> Int -> Int
            add number1 number2 = number1 + number2

            -- Let's curry it!

            add5 : Int -> Int
            add5 = add 5

            -- Let's use it now!

            marks : List Int
            bumpedMarks = List.map add5 marks

Here is how to extract sub-functions but we can also pipe results

Currying - real example

          
            weatherForecast : LinearRegression
            weatherForecast = List.map .weather lastYearDays     -- List Day           -> List Weather
              |> List.map .topTemperature                        -- List Weather       -> List Int
              |> List.indexedMap (\index temp -> [index, temp] ) -- List Int           -> List [ Int, Int ]
              |> computeLinearRegression 2                       -- List [ Int, Int ]  -> LinearRegression

In this example, I compute a linear regression of the temperatures in Costa Rica

A sound type-system

  • Strong
  • Structural
  • No null
  • Implicit OR explicit

Another place where Elm shines is the type system

  • Strong: type known in whole codebase
  • Structural: type defined by structure (duck typing)
  • No null: places where data can miss are known and handled with Maybe => all edge cases handled by design
  • Implicit or explicit: whatever fits the local needs

The flux pattern

Uni-directional data flow

The app is everything above elm runtime

  • The model that represents the WHOLE app is transformed in a FULL HTML
  • The elm runtime does DOM things
  • The user creates Msg via interactive elements
  • The Msg are used to update the model

The DOM is generated by Elm via virtual DOM and diffing

The diff is way more powerful in elm than JS because of elm's guarantees

The elm runtime contains ALL the non-pure functions (connections + randomness)

Cost of performance

Two techniques

  1. Lazy
  2. Indexed

Spent 5min getting to 100 on lighthouse performance. Never so low on a JS app.

For app reactivity, reduce DOM manipulations further with:

  • Lazy: don't try to generate the HTML when input hasn't changed
  • Indexed: avoid index shifting by pointing Elm to elements ids

Haven't used but seems WAY simpler than JS alternatives (React and AngularJS)

            urlToPage url =
              url
                |> Url.parse urlParser
                |> Maybe.withDefault Index

              Url.oneOf
                [ Url.map Index Url.top                   -- Matches /

Serialization/Deserialization 

          
            type alias StorageAppState =
              { appKey: String
              , accessToken: Maybe String
              , selectedLanguage: String
              }

            storageAppStateDecoder : Decoder StorageAppState
            storageAppStateDecoder =
              Decode.succeed StorageAppState
                |> Pipeline.optional "appKey" Decode.string ""
                |> Pipeline.optional "accessToken" (maybe Decode.string) Nothing
                |> Pipeline.optional "selectedLanguage" Decode.string "defaultLanguage"

            decodeStorageAppState : Decode.Value -> StorageAppState
            decodeStorageAppState appKeyAsJson =
              case Decode.decodeValue storageAppStateDecoder appKeyAsJson of
                Ok appState -> appState
                Err _ -> StorageAppState "" Nothing "defaultLanguage"

HTTP 

          
            fetchAllStats : String -> String -> Cmd Msg
            fetchAllStats sheetId accessToken = Http.request
              { method = "GET"
              , headers = [ createBearerHeader accessToken ]
              , url = computeSheetDataUrl sheetId
              , body = Http.emptyBody
              , expect = Http.expectString (StatsMsg << GotStats)
              , timeout = Nothing
              , tracker = Nothing
              }

            update: Msg -> Model -> (Model, Cmd Msg)
            update msg model =
              case statsMsg of
                GotStats httpResult ->
                  case httpResult of
                    Ok statsAsString -> updateStats statsAsString model eraseAppKeyFromStorage
                    Err error ->
                      case error of
                        BadStatus status ->
                          case status of
                            401 -> ( model, load ( makeAuthorizationUrl model.baseUrl ) )
                            403 -> ( model, load ( makeAuthorizationUrl model.baseUrl ) )
                            _ -> ( model, Cmd.none )

                        _ -> ( model, Cmd.none )

Verbose (all fields required) but simple

Must go through Elm runtime because un-pure

Fire and forget
